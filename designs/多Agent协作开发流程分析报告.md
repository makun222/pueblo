# 多Agent协作开发流程分析报告

## 总体评价

这是一个**有清晰愿景但实践上有重大隐患**的流程设计。它借鉴了传统软件工程中的"阶段门禁（Stage-Gate）"模型，引入了多层评审来对抗LLM幻觉，意图很好。但它忽略了**Agent协作的独特成本结构**——传统工程中"开会评审"的代价是"人时"，在Agent世界里代价是"token + 上下文窗口 + 推理质量衰减"。下面逐点分析。
1.phase plan： 
1.1做方案:agentA 规划需求方案（细化需求，交互，评估现有资源，可行性，需澄清的细节） 
1.2审方案:agentB 按照方案，检查实际代码情况，杜绝幻觉 
1.3改方案： 
1.4再审。 
1.5重复1.3-1.4，直到方案无误，或超过本轮预算。 
1.6用户确认。或澄清并打回1.3。或追加预算。 
2.phase design： 
2.1设计:agentC制定设计方案（接口调用、前端设计、交互实现） 
2.2设计评审:agentD评审设计方案（需求条目覆盖度；冲突、影响、性能）                     agentE评审设计方案（需求覆盖与匹配度、设计目标满足度） 
2.3设计修改:agentC按照D、E意见修改设计方案。 
2.4再评审。 
2.5重复2.2-2.3，直到评审通过，或超过本轮预算。 
2.6设计分解：agentC对方案进行分解（分解任务书（短上下文），任务编排，N个agent各自实现） 
2.7分解评审：agentD评审（任务书冲突；任务书与设计方案匹配度，编排冲突） 
2.8重复2.6-2.7，直到评审通过，或超过本轮预算。 
2.9测试用例：agentE设计测试用例 
2.10测试评审：agentF评审测试用例（测试有效性、测试覆盖度） agentC、D评审测试用例（是否符合程序设计） agentA、B评审测试用例（测试目标是否与需求条目一致） 
3.实现 
3.1代码实现：agentC调度agentC1/agentC2/agentC3/...agentCN实现任务书 
3.2单元测试：agentC1/agentC2/agentC3/...agentCN各自开展单元测试并修复 
3.3代码合并：agentC合并代码 
3.4联调测试：agentE运行联调测试用例 
3.5代码修改：agentC判断bug，调度agentC1/agentC2/agentC3/...agentCN修改任务书代码，agentC修改整体性代码 
3.6代码review：agentC、agentD review代码，提出问题。重复3.5。 
3.7测试回归：agentE回归代码。重复3.5-3.6，直到无新增代码问题。或达到本轮预算。
4.发布 代码提交git。

## 一、核心问题分析

### 问题1：Agent角色过载且职责边界模糊

| Agent | 各阶段职责 | 问题 |
|-------|-----------|------|
| **C** | 方案设计 → 改设计 → 分解任务 → 调度实现 → 合并代码 → 判断Bug → Review代码 | **单点故障**。C身兼架构师、项目经理、技术主管、代码合并者、Bug裁判5个角色，在任何子任务上的质量衰减会级联影响下游 |
| **E** | 评审设计(需求匹配度) → 设计测试用例 → 回归测试 | **角色冲突**。E既设计测试又执行测试，失去了测试独立性；同时E在2.2是评审者、2.9是设计者、3.4是执行者，上下文切换成本高 |
| **A/B** | 方案制定、方案评审 → (跨阶段)评审测试用例 | 跨阶段参与是好的，但A/B在2.10突然出现评审测试用例，缺乏连续上下文 |

**影响**：角色职责不纯，评审独立性受损，故障时难以定位责任链。

### 问题2：评审链条过长，上下文衰减严重

从设计到可执行代码的路径（简化版）：

```
2.1 设计(C) → 2.2 评审(D,E) → 2.3 改设计(C) → 2.4 再评审(D,E) [循环]
→ 2.6 分解(C) → 2.7 评审(D) [循环]
→ 2.9 测试用例(E) → 2.10 评审(F, C, D, A, B) [循环]
→ 3.1 实现(CN) → 3.2 UT(CN) → 3.3 合并(C) → 3.4 集成测试(E)
→ 3.5 修复(C, CN) → 3.6 Review(C, D) [循环]
→ 3.7 回归(E) [循环]
```

一个需求从提出到落地，至少经过 **7层评审门禁**。在传统工程中这可以接受（每个门禁是30分钟会议），但在Agent模型中，每层评审都是一次完整的LLM调用+上下文加载，累积的**信息衰减**会导致：
- 下游Agent读到的设计文档越来越"失真"
- 循环次数等于token消耗的指数增长
- 最终代码质量取决于距离原始需求最远的那一跳

### 问题3：退出的条件不可度量

多个地方使用了"直到评审通过"这个条件。但什么是"通过"？

- D和E都同意算通过？还是任意一个反对就驳回？
- D和E意见冲突时谁裁决？
- "超过本轮预算"作为唯一的兜底条件，意味着**质量可以被预算随意截断**，且没有"部分接受"的中间状态

### 问题4：缺少变更反馈回路

代码实现阶段发现的架构问题（3.4联调失败）如何处理？

```
需求 → 方案 → 设计 → 分解 → 实现 → 联调失败
                                     ↓
                                 发现架构缺陷
                                     ↓
                                 应该回到哪里？
```

你的流程没有定义**跨阶段回退机制**。如果3.4发现是设计方案的问题，应该回到2.1还是修改当前设计？这会影响到整个流程的鲁棒性。

### 问题5：Phase 1的"杜绝幻觉"过于理想化

> 1.2 审方案: agentB 按照方案，检查实际代码情况，杜绝幻觉

让一个Agent去检查另一个Agent的工作是否存在幻觉，本质上是用**同样的LLM能力**去做验证。除非B有A没有的信息源（如实际的代码仓库执行结果），否则这只能减少表面幻觉（如文件名拼错），无法消除**概念幻觉**（如虚构了不存在的API）。

## 二、可保留的强设计元素

尽管有上述问题，你的设计中有几个非常值得保留的核心理念：

### ✅ 多视角评审矩阵

在2.2和2.10中，你设计了**不同关注维度的评审者**（D关注冲突/性能，E关注需求匹配度，A/B关注需求条目覆盖），这个**正交评审**（Orthogonal Review）的思路是对的——关键在于不要串行化，而是让它们异步并行。

### ✅ 任务分解的显式化

2.6-2.8将设计方案分解为可并行执行的"任务书"，这是重要的抽象层次转换。每个子任务上下文短、职责单一，适合独立的Agent执行——这是符合LLM上下限约束的正确做法。

### ✅ 预算感知的循环控制

"超过本轮预算就停止"是一个务实的兜底策略，只是需要补充"降级接收条件"（见下文建议）。

## 三、基于Pueblo平台的优化建议

结合Pueblo的现有模式（Agent模板、计划模板、Constitution检查），我建议你重构为以下结构：

### 建议架构：缩减层数 + 明确制品 + 异步评审

```
Phase 1: 方案与设计（合并Plan+Design）
├── 1.1 AgentA（需求负责人）产出：需求制品（spec.md）
│    ├── 需求条目（可测试、可追踪）
│    └── 验收标准（AC）
├── 1.2 AgentB（技术负责人）产出：架构方案（plan.md）
│    ├── 基于实际代码库的分析
│    ├── 模块边界与接口定义
│    └── 约束与风险登记
├── 1.3 并行Constitution检查（使用Pueblo现有的模板）
│    └── 自动检查：模块边界、接口定义、并行任务识别、测试规划...
├── 1.4 用户确认 + 澄清 → 进入实现阶段

Phase 2: 实现
├── 2.1 AgentB 分解任务 → tasks.md（Pueblo现有的tasks.md格式）
│    └── 每个任务：上下文短、产出明确、可独立评审
├── 2.2 并行实现（AgentC1..CN各自执行任务书）
├── 2.3 AgentB 合并代码
├── 2.4 AgentE（独立测试Agent）运行集成测试
├── 2.5 缺陷修复循环（AgentB调度）
└── 2.6 代码Review（AgentD作为独立Reviewer，只做一次，非循环）

Phase 3: 发布
└── Git提交 + 标签
```

### 具体的8项改进建议

| # | 问题 | 建议 |
|---|------|------|
| 1 | **评审门禁过多** | 将4层评审（方案、设计、分解、测试）合并为**2层**：方案评审（含需求+设计）和交付评审（含代码+测试）。每层只有1轮评审+1轮修改 |
| 2 | **Agent C过载** | 拆分角色：**TechLead（B）** 负责架构决策和分解；**Implementer（C1..CN）** 负责实现；**Merger（M）** 只做代码合并和冲突解决，不做设计判断 |
| 3 | **评审通过标准模糊** | 为每个制品定义**评审清单**（Checklist），以"是/否"可判定问题替代主观判断。例如不是"设计是否合理"而是"每个模块接口是否有入参/出参/错误码定义" |
| 4 | **缺少降级接收** | 增加 **AC分类**：P0（必须满足）/P1（尽量满足）/P2（锦上添花）。预算不足时P0必须通过，P1/P2可以降级 |
| 5 | **缺少回退机制** | 定义明确的**回退级别**：实现中发现的设计缺陷→回到2.1（不重新评审需求）；需求理解偏差→回到1.1；Bug→留在3.x循环 |
| 6 | **"杜绝幻觉"方式不当** | 改为**基于执行的验证**：Agent B应该运行工具（如grep/glob/exec）查询实际代码库，而不是"阅读方案后判断"。执行结果比LLM判断更可靠 |
| 7 | **测试独立性不足** | Agent E的职责限定在**测试执行和结果报告**，测试用例由独立Agent F设计。或者使用Pueblo的debugger模板作为测试执行者 |
| 8 | **上下文连续性问题** | 每次阶段切换时，使用**制品摘要**（artifact summary）传递关键信息，而不是完整上下文。例如方案评审通过后，只把"接口定义+约束条件"传给实现阶段，而不是完整对话历史 |

## 四、可行性判断

**部分可行，需要重构。**

- **核心理念（多视角评审、任务分解、预算控制）**：✅ 可行，且与Pueblo的现有模式（Constitution检查、Task分解）兼容度很高
- **具体执行流程（7层串行门禁、角色过载、无回退）**：❌ 在实践上会因上下文衰减和token爆炸而快速崩溃

**推荐的实施路径**：
1. 先用**简化版**（2个阶段、2层评审、角色职责纯净化）跑通一个迭代
2. 根据实际token消耗和代码质量数据，逐步增加评审维度
3. 用Pueblo的Constitution检查替代人工Agent评审中的常规检查项，节省评审Agent的token用于更关键的判断

## 如何使用

### 1. 初始化一个项目
```bash
cd your-project-root
python skills/pueblo-dev-flow/scripts/project-init.py .
```

### 2. 执行编排
按 SKILL.md 中的 Pipeline Orchestration 步骤依次执行。每个步骤直接引用了角色卡片和模板。

### 3. 关键操作模式
- **BLOCKING** 步骤 → 等待用户确认（如 Phase 1 结束时的用户确认）
- **GATE 条件** → 自动化判断 PASS/REVISE/FAIL，不等待用户
- **预算保护** → 通过 scripts/budget-track.py 追踪消耗

### 4. 追踪预算
```bash
# 查看状态
python skills/pueblo-dev-flow/scripts/budget-track.py . status

# 消耗预算 (Phase 1, 消耗3次)
python skills/pueblo-dev-flow/scripts/budget-track.py . consume 1 3

# 追加预算 (Phase 2, 追加10次)
python skills/pueblo-dev-flow/scripts/budget-track.py . add-budget 2 10
```

## 设计哲学摘要

这个编排的本质是 **"用制品的确定性来对抗LLM的非确定性"**。每个阶段产出的制品（PRD、设计文档、接口契约、任务书、测试用例）都是后续阶段的唯一输入，Agent之间的交流不依赖对话历史，而依赖文件系统和明确的契约引用。评审的目的是验证"信息在传递中没有衰减"，而非"谁的对错"。
