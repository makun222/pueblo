# Pueblo Constitution

## Core Principles

### I. 代码库优先与模块化架构
代码库 MUST 以模块为中心组织，模块内部 MUST 保持高内聚，模块之间 MUST 保持低耦合。
每项新增能力 MUST 归属到明确的模块边界，并提供可审查的接口定义。模块契约 MUST
支持后续扩展，避免因局部变更引发跨模块级联修改。理由：以代码库为核心组织方式，
可以在功能增长时维持可维护性、可扩展性和多 agent 并行开发能力。

### II. CLI 优先与简洁前端
产品主交互方式 MUST 为命令行界面。若提供前端界面，界面 MUST 保持简洁、聚焦任务，
不得以视觉复杂度替代功能清晰度。输入区与输出区 MUST 支持动态增减和尺寸调整，以适
配多输入、多输出及对比分析场景。理由：CLI 适合工程化操作与自动化集成，简洁前端则
为复杂输入输出任务提供必要辅助而不牺牲效率。

### III. 需求清晰与关键流程可视化
需求描述 MUST 清晰、可实现、可验证，并明确目标、边界和验收条件。重点需求 MUST
提供时序图和用例图，用于说明关键交互路径、参与者职责和模块协作关系。规格说明 MUST
在进入实施规划前明确模块范围、外部接口与验收边界。理由：清晰、可视化的需求是准确
拆解任务、减少返工和支持多 agent 协同的前提。

### IV. 迭代交付与并行任务拆解
工作 MUST 按迭代推进，每次迭代 MUST 完成交付一个或多个模块需求的完整闭环。模块
设计 MUST 先整理功能清单、接口设计与开发任务拆解，再进入编码实现。实施计划 MUST
明确可由多个 agent 并行执行的任务边界、依赖关系与合并顺序。理由：先定义模块能力与
接口，再拆解任务，才能在保持架构稳定的前提下安全并行开发。

### V. 必要功能优先、测试驱动与每日版本提交
产品设计与实现 MUST 聚焦必要功能，only necessary。任何新增能力 MUST 证明其直接服务
当前迭代目标、核心用户价值或关键质量门禁；不能证明必要性的能力 MUST 延后，不得为
“可能以后会用到”而提前建设。开发过程 MUST 采用测试驱动开发，先定义测试，再实现功
能，并通过重构收敛设计。每次迭代交付 MUST 包含集成测试，用于验证本次完成的模块切
片、接口契约和关键协作流程。代码库 MUST 纳入版本控制，所有活跃开发中的工作成果 MUST
至少每日提交一次。理由：必要功能优先可抑制范围膨胀，测试驱动和集成交付可控制质量风
险，每日提交则保障可追溯性与恢复能力。

## 设计与文档标准

每个模块设计文档 MUST 列出功能清单、对外接口与内部接口、依赖关系、耦合假设以及覆
盖单元测试与集成测试的测试策略。功能清单 MUST 区分“本迭代必要功能”与“后续扩展候
选”，不得将未证明必要性的内容混入当前交付范围。重点需求文档 MUST 包含时序图和用例
图。涉及 UI 的规格文档 MUST 说明输入区与输出区的动态增减行为、尺寸调整规则及约束条
件。若存在刻意延后定义的信息，MUST 使用带原因说明的 `TODO(...)` 标记，禁止使用无解
释占位符。

## 交付流程与质量门禁

迭代范围 MUST 控制在可独立评审、可独立验收的模块成果内。规划产物 MUST 按模块组织
工作，并标注适合多 agent 并行执行的任务。每次迭代 MUST 优先交付必要功能，不得将可
选增强、预研性质能力或未来扩展接口伪装为当前必需项。接口定义与任务拆解完成前，实
施工作 MUST NOT 开始。测试驱动证据 MUST 在实现任务前可追踪，集成测试未通过前，迭
代 MUST NOT 视为完成。所有活跃工作流 MUST 遵守每日提交要求，避免长期未提交导致协
作失真。

## Governance

本宪章高于其他本地流程说明；若存在冲突，以本宪章为准。

任何修订 MUST 同步更新宪章正文、同步影响报告及关联模板，完成对齐后方可视为修订完
成。

宪章版本号 MUST 采用语义化版本规则：`MAJOR` 用于删除原则或对原则作出不兼容重定义；
`MINOR` 用于新增原则或显著扩展治理要求；`PATCH` 用于澄清、措辞优化与非语义性修订。

每份 `plan`、`spec` 与 `tasks` 产物 MUST 执行显式的宪章符合性检查。符合性检查 MUST
覆盖模块内聚/耦合、CLI 优先与前端简洁性、需求完整性、时序图与用例图要求、迭代范围、
必要功能边界、并行任务拆解、测试驱动证据、集成测试覆盖与每日提交纪律。

若首次采纳日期无法追溯，MUST 使用显式 TODO 保留并在后续治理中补齐，不得伪造历史日
期。
