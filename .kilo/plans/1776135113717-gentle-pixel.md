### Core Principles

1. `I. 代码库优先与模块化架构`
   - 代码库结构 MUST 以模块为中心组织，模块内高内聚、模块间低耦合。
   - 每项新增能力 MUST 归属到明确的模块边界，并具备已记录的接口定义。
   - 模块契约 MUST 支持后续扩展，避免因单点变更引发跨模块级联修改。
   - 理由：保证代码库可维护、可扩展，并支持并行开发。

2. `II. CLI 优先与简洁前端`
   - 产品主交互方式 MUST 为命令行界面。
   - 若提供前端界面，界面 MUST 保持简洁、聚焦任务，不引入无必要复杂度。
   - 输入区与输出区 MUST 支持动态增减与尺寸调整，以适配多输入、多输出场景。
   - 理由：兼顾操作效率与交互灵活性，降低界面复杂度。

3. `III. 需求清晰与关键流程可视化`
   - 需求描述 MUST 清晰、可实现、可验证，避免含糊措辞。
   - 重点需求 MUST 提供时序图和用例图。
   - 规格说明 MUST 在进入实施规划前明确模块范围、外部接口与验收边界。
   - 理由：降低歧义，提高评审效率，并为多 agent 协作提供稳定输入。

4. `IV. 迭代交付与并行任务拆解`
   - 工作 MUST 按迭代推进，每次迭代 MUST 交付一个或多个模块需求的完整闭环。
   - 模块设计 MUST 先形成功能清单、接口设计与开发任务拆解。
   - 实施计划 MUST 明确可由多个 agent 并行执行的任务边界与依赖关系。
   - 理由：保证迭代可交付、任务可拆分，并降低并行开发冲突。

5. `V. 测试驱动、集成交付与每日版本提交`
   - 开发过程 MUST 采用测试驱动开发。
   - 每次迭代交付 MUST 包含集成测试，用于验证本次完成的模块切片。
   - 代码库 MUST 纳入版本控制，活跃开发中的工作成果 MUST 至少每日提交一次。
   - 理由：保证交付质量、及时暴露回归问题，并保持可追溯的项目历史。

  - 覆盖单元测试与集成测试的测试策略

- 涉及 UI 的规格文档 MUST 说明输入/输出区域动态增减与尺寸调整行为。

### Section 3

Title: `交付流程与质量门禁`

Content to include:

- 迭代范围 MUST 控制在可独立评审、可独立验收的模块成果内。
- 规划产物 MUST 按模块组织工作，并标注适合多 agent 并行执行的任务。
- 在接口定义与任务拆解完成前，实施工作 MUST NOT 开始。
- 集成测试未通过前，迭代 MUST NOT 视为完成。
- 所有活跃工作流 MUST 遵守每日提交要求。
