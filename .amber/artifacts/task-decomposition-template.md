# 任务分解 模板

## Task Manifest — 任务清单与编排

### 元数据
```
version: 1.0
author: Agent C (Architect)
basedOnDesign: architecture.md v{version}
totalTasks: {N}
```

### 任务依赖图 (拓扑排序)

```
任务依赖图谱（文本 DAG）:

task-01 (模块A-基础结构)
  └── requiredBy: task-02, task-03
task-02 (模块A-核心逻辑)
  └── dependsOn: task-01
  └── requiredBy: task-04
task-03 (模块B-接口层)
  └── dependsOn: task-01
  └── requiredBy: task-05
...
```

### 任务编排执行顺序

```
Phase 1 (并行):
  → task-01, task-03

Phase 2 (依赖就绪):
  → task-02 (等待 task-01)
  → task-04 (等待 task-02)

Phase 3:
  → task-05 (等待 task-03, task-04)
```

## 单个任务书模板

`decomposition/tasks/task-{NN}.md`

```markdown
# Task-{NN}: {任务标题}

## 元数据
- 所属模块: {模块名}
- 设计引用: [DESIGN-{段落}] [INTERFACE-{接口名}]
- PRD引用: [PRD-{N}]
- 预估复杂度: [S | M | L]
- 依赖任务: task-{XX} (前置依赖)
- 被依赖任务: task-{YY} (后置依赖)

```

{函数签名 / API端点 / 组件接口}

从上游接口读取的数据模型:

{数据模型名称及字段}
```

```

## 任务书质量控制规则

在输出任务书前，C要逐项检查：
- [ ] 任务书总长度 < 2000 token
- [ ] 输入契约引用了已存在的接口定义
- [ ] 输出契约与下游任务的输入一致
- [ ] 依赖关系已在前置/后置任务字段中标注
- [ ] 验收标准包含正常、边界、异常三种场景
- [ ] 技术提示给出了实现方向但不过度约束实现细节
