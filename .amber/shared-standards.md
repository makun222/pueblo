# 全局共享标准

本文档定义了所有参与pueblo-dev-flow的Agent必须遵守的通用规则。

---

## 1. 交流规范

### 1.1 制品优先原则
- 所有重要输出必须是文件制品（写入`artifacts/`目录）
- 对话中的口头承诺不计入正式决策
- 任何评审意见必须标注：问题位置、严重等级、建议修改方案

### 1.2 引用可追溯
- 每个制品中的陈述须引用上游来源（PRD条目、接口签名、设计段落）
- 格式示例：`[PRD-2.3]` 或 `[DESIGN-interface-auth]`
- 无引用的断言默认为Agent自身推理，需标注 `[INFER]`

### 1.3 不确定性的表达
当Agent对信息不确定时，必须明确标注信心等级：
- `[CONFIRMED]`: 已从上游制品验证
- `[HIGH]`: 推理可靠，但未找到直接引用
- `[MEDIUM]`: 推理合理，但可能有其他解释
- `[LOW]`: 推测，需要更多信息

---

## 2. 输出质量规范

### 2.1 制品必须自包含
每个制品文件应能在不查阅对话历史的情况下独立理解：
- 文件头部注明：来源、版本、最后更新Agent
- 正文包含必要的上下文摘要
- 外部引用使用可解析路径

### 2.2 接口契约规范 (Interface Contracts)
- 每个公开API/函数必须包含类型签名、前置条件、后置条件
- 数据模型定义字段名、类型、约束、示例值
- 事件/消息定义payload结构和语义

### 2.3 评审意见分级
| 等级 | 含义 | 必须修复 | 影响 |
|------|------|---------|------|
| P0 | 架构违规/安全漏洞/关键功能缺失 | 是 | 阻断门禁PASS |
| P1 | 非关键功能缺陷/规范偏离 | 是 | 阻断门禁PASS |
| P2 | 建议改进/代码风格 | 否 | 记录但不阻断 |
| P3 | 待确认的疑问 | 否 | 需回答但不阻断 |

---

## 3. 协作规则

### 3.1 输入读取原则
- 读取上游制品时，优先读完整内容而非摘要
- 涉及大型文件时，使用针对性搜索（grep）而非整体加载
- 上游制品的更新版本应标记日期/版本号

### 3.2 状态不可变原则
- 已评审通过的制品不允许后续阶段单方面修改
- 如需修改，走"跨阶段变更流程"
- `.project/phase-state.json` 是唯一的状态权威来源

### 3.3 上下文预算纪律
- 每个Agent的单次交互应控制在上下文窗口的70%以内
- 任务书控制在2000 token以内
- 评审意见聚焦高优先级问题（P0/P1），避免铺陈

---

## 4. 失败处理

### 4.1 Gate FAIL 处理
当一个GATE条件返回FAIL时：
1. 记录失败原因到 `.project/phase-state.json` 的 `failures` 数组
2. 判断失败性质：
   - **可恢复**: 属于实施偏差（如任务书写错），回到对应步骤修正
   - **不可恢复**: 属于上游缺陷（如PRD矛盾），需回退到上游阶段
3. 每一步FAIL消耗预算（计入 `budget.json`）

### 4.2 评审僵局处理
当GATE反复REVISE（超过3轮）仍不收敛：
1. 自动生成分歧摘要报告
2. 标记 `[ESCALATED]` 到 `.project/phase-state.json`
3. 暂停评审，等待用户裁决

---

## 5. 预算管理

### 5.1 预算单位
预算以"Agent调用次数"计量。每调用一次Agent消耗1单位。

### 5.2 阶段预算分配
| Phase | 默认预算 | 用途 |
|-------|---------|------|
| Phase 1 Plan | 15 次 | PRD+评审+修改循环 |
| Phase 2 Design | 25 次 | 设计+评审+分解+测试计划 |
| Phase 3 Implement | 取决于任务数 | 实现+UT+集成+Review循环 |
| Phase 4 Release | 5 次 | 最终检查+提交 |

### 5.3 预算超额处理
超额时自动通知用户，提供选项：
- 追加预算
- 截断到当前阶段已完成的核心功能
- 降级评审深度（跳过P2/P3评审）
